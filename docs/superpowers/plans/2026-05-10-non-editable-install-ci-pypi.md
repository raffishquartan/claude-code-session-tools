## Task 2: Replace `.github/workflows/ci.yml`

**Files:**
- Modify: `.github/workflows/ci.yml`

- [ ] **Step 1: Replace ci.yml with the following content**

  ```yaml
  name: CI

  on:
    push:
      branches: [main]
    pull_request:
      branches: [main]

  jobs:
    test:
      runs-on: ubuntu-latest
      strategy:
        fail-fast: false
        matrix:
          python-version: ["3.10", "3.11", "3.12", "3.13"]
      steps:
        - uses: actions/checkout@v4

        - uses: astral-sh/setup-uv@v4
          with:
            python-version: ${{ matrix.python-version }}

        - name: Install package and dev deps (non-editable)
          run: uv sync --no-editable --extra dev

        - name: Run tests
          run: uv run pytest -q

    install-check:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v4

        - uses: astral-sh/setup-uv@v4

        - name: Install via uv tool (non-editable, production path)
          run: uv tool install .

        - name: Add uv tool bin to PATH
          run: echo "$(uv tool dir --bin)" >> $GITHUB_PATH

        - name: Verify all CLIs report versions
          run: |
            ccd --version
            ccr --version
            ccs --version
            claude-code-usage --version
  ```

  The `install-check` job is the direct guard against the failure mode we fixed:
  it exercises `uv tool install .` (non-editable production path) and verifies
  all four CLIs work. If a worktree-based install ever slips in again, this job
  catches it on the next CI run.

## Task 7: Create `.github/workflows/release.yml`

**Files:**
- Create: `.github/workflows/release.yml`

- [ ] **Step 1: Create `release.yml`**

  Write `.github/workflows/release.yml` with the following content:

  ```yaml
  name: Release

  on:
    push:
      tags:
        - 'v*'

  permissions:
    contents: write   # create GitHub Release and upload assets
    id-token: write   # OIDC token for PyPI trusted publisher

  jobs:
    release:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v4

        - uses: astral-sh/setup-uv@v4

        - name: Build wheel and sdist
          run: uv build

        - name: Create GitHub Release and upload assets
          uses: softprops/action-gh-release@v2
          with:
            files: dist/*
            generate_release_notes: true

        - name: Publish to PyPI
          uses: pypa/gh-action-pypi-publish@release/v1
  ```

  Notes on the design:
  - `permissions.contents: write` is required for `softprops/action-gh-release` to
    create the release.
  - `permissions.id-token: write` is required for OIDC. `pypa/gh-action-pypi-publish`
    picks up the OIDC token automatically — no API key or secret is stored.
  - `generate_release_notes: true` auto-populates release notes from merged PRs since
    the previous tag.
  - PyPI publish runs after GitHub Release. If PyPI fails the GitHub Release still
    exists, so assets are never lost.
  - This workflow does NOT fire until a `v*` tag is pushed. Creating it in this PR is
    safe — it is dormant until you explicitly release.
